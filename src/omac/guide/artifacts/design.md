# design 产物合同

## 使用场景

本合同用于 `plan` 产出或评审阶段，约束设计文档如何成为后续验收、DAG 拆解和实现的稳定依据。

第一步必须运行：

```bash
omac work show <issue-id> --output json
```

以返回的 task、context、authority、guide_refs 和 submit 为当前实例事实。本文是静态 guide，
只能补充长期格式与校验规则，不能覆盖实例事实、contract、上轮评审或精确提交命令。

## 最小合法示例

设计文档使用 Markdown 正文 + YAML frontmatter：

```md
---
schema: omac.design/v1
title: 登录会话续期
problem: 旧会话过期后用户会无提示退出
non_goals:
  - 不重构无关支付流程
flows:
  - flow-login-renewal
risk_level: medium
---

# 设计文档

## 背景

现有会话过期后直接返回未认证，用户输入可能丢失。

## 目标与非目标

续期成功时保留当前操作；续期失败时进入现有登录流程。

## 业务流程

`flow-login-renewal`：请求遇到会话过期 → 尝试续期 → 重放一次原请求。

## 核心数据

- Session：认证模块所有；登录时创建，续期时更新，退出时删除。
- 边界条件：空令牌、重复续期、权限不足和旧会话数据均有明确结果。

## 模块边界

- `auth` 拥有会话状态；业务模块只调用认证接口，不直接修改会话。

## 跨模块契约

- 输入：过期会话；输出：新会话或稳定错误；原请求最多重放一次。

## 风险与兼容性

- 保留现有登录、退出和权限拒绝行为；续期失败不改变原错误语义。

## 验收映射

- `flow-login-renewal` → 会话续期流程、失败回退和对应验证入口。
```

## 字段语义

frontmatter 是机器与后续 Agent 的结构锚点，正文负责解释实现意图：

| 字段或章节 | 语义 |
|---|---|
| `schema` | 固定为 `omac.design/v1`。 |
| `title` | 可独立识别的方案名称。 |
| `problem` | 要解决的真实生产问题，不写抽象愿景。 |
| `non_goals` | 明确相邻但不在本次范围内的事项。 |
| `flows` | 引用验收文档中的稳定 flow id。 |
| `risk_level` | 标明改动风险，正文解释风险来源与缓解方式。 |
| 背景、目标与非目标、业务流程 | 说明为什么做、做到什么程度以及用户/系统如何流转。 |
| 核心数据 | 写清字段、状态、所有者，以及创建、更新、删除路径，避免不必要的数据复制或转换。 |
| 模块边界 | 写清依赖方向、主要修改路径和禁止修改路径。 |
| 跨模块契约 | 写清调用方向、DTO、事件、输入输出、错误码或异常语义和状态变化。 |
| 风险与兼容性 | 列出现有行为、调用方、旧数据和回滚路径，保证不破坏用户空间。 |
| 验收映射 | 把每个 flow 映射到设计章节和可执行验证入口。 |

后续执行者可能是低推理预算模型。文档必须显式给出边界条件，不能要求执行者自行补全
隐含上下文。空输入、重复提交、失败回滚、权限不足和旧数据兼容等处理方式应落到具体章节。
不强制采用特定方法论；领域术语不能替代具体数据、边界和契约。

## 校验硬门

1. 当前 `plan` submit validator 首先检查文件存在且非空；通过这一机器门不代表语义合格。
2. frontmatter 必须声明 `schema: omac.design/v1`，并给出问题、非目标、flows 和风险等级。
3. 业务流程必须能映射到验收文档 flow，且 `验收映射` 不得遗漏 frontmatter 中的 flow。
4. 核心数据必须说明所有权和修改路径；模块边界必须说明依赖方向。
5. 跨模块契约必须覆盖输入输出、错误和状态；风险与兼容性必须说明受影响的现有行为。
6. 所有边界条件都必须有确定处理方式，不能留给 worker 猜测。

## 常见错误 → 修正

| 常见错误 | 修正 |
|---|---|
| 只有架构名词，没有数据所有权 | 为每个核心数据补字段、状态、所有者和生命周期。 |
| 把异常路径写成“实现时决定” | 明确空输入、重复请求、回滚、权限和旧数据的结果。 |
| 模块边界只列目录 | 补依赖方向、允许修改路径、禁止修改路径和跨模块接口。 |
| flow 只出现在正文且名称漂移 | 在 frontmatter 使用稳定 flow id，并逐项写验收映射。 |
| 以理论正确为由改变旧行为 | 在风险与兼容性中列出现有行为并给出零破坏方案。 |

## 提交

提交前重新读取 `work show`，使用其返回的精确 submit 命令。`plan` 产出的常见形状是：

```bash
omac work submit <issue-id> --plan-file <file>
```

不要在产出阶段提交 verdict，也不要手动推进平台状态。
