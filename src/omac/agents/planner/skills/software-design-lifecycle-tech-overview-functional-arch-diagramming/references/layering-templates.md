# 系统功能架构图常见分层模板库

这份参考文件是 `system-functional-architecture-diagramming` 的配套模板库。

用途不是替代具体设计，而是给架构师在概要设计阶段提供**可直接套用、再按场景裁剪**的分层骨架。每个模板都回答同一个问题：

- 系统建设目标大致是什么
- 功能应该按什么轴来分层
- 哪些节点通常放在核心闭环
- 哪些能力应该作为支撑/治理能力出现
- 什么内容不应该混进系统功能架构图

---

## 使用方法

拿到一个新系统时，先不要直接画。

先做三步：

1. 先写一句建设目标
   - 本系统的建设目标是：______

2. 判断核心问题更像哪一类
   - 交互驱动
   - 业务处理驱动
   - 平台能力驱动
   - 数据加工驱动
   - 治理/风控驱动

3. 从下面选一个最接近的模板作为骨架
   - 先套骨架
   - 再替换节点名
   - 最后删除不必要层级

重要原则：
**模板是起点，不是标准答案。不能为了套模板而扭曲系统本身。**

---

## 模板 1：标准业务系统五层模板

### 适用场景

适用于大多数有明确用户入口、应用编排、核心业务能力、治理支撑、数据协同的系统。

例如：
- 风控平台
- 订单系统
- 内容生产平台
- 任务处理平台
- 资金安全校验平台

### 推荐分层

1. 接入与交互层
2. 应用编排层
3. 核心业务能力层
4. 支撑与治理层
5. 数据与外部协同层

### 各层典型节点

#### 1) 接入与交互层
- 用户入口
- 管理后台
- 开放接口
- 消息接入
- 批量触发

#### 2) 应用编排层
- 场景编排
- 流程调度
- 指令分发
- 任务协调

#### 3) 核心业务能力层
- 规则执行
- 风险校验
- 订单处理
- 内容生成
- 结果判定

#### 4) 支撑与治理层
- 配置管理
- 规则管理
- 权限控制
- 审计留痕
- 监控告警

#### 5) 数据与外部协同层
- 数据汇聚
- 指标服务
- 主数据读取
- 外部系统适配
- 结果回传

### 典型核心闭环

`用户/系统触发 → 应用编排 → 核心能力执行 → 结果输出/回传`

### 什么时候不要用

如果系统本质不是“标准业务系统”，而是明显的平台中枢、数据流水线或治理中台，不要硬套这个模板。

---

## 模板 2：平台型系统中心能力模板

### 适用场景

适用于一个中心平台向多个业务方、多个系统、多个场景输出能力的系统。

例如：
- 统一规则平台
- API 网关平台
- 统一鉴权平台
- 算法服务平台
- 企业服务总线型平台

### 推荐分层

1. 多端接入层
2. 平台接入与路由层
3. 中心能力层
4. 平台治理层
5. 外部协同与数据支撑层

### 各层典型节点

#### 1) 多端接入层
- 业务系统接入
- 运营入口
- 开放方接入
- 内部管理端

#### 2) 平台接入与路由层
- 请求接入
- 鉴权校验
- 路由分发
- 流量控制

#### 3) 中心能力层
- 规则执行
- 能力编排
- 统一决策
- 公共服务封装

#### 4) 平台治理层
- 配置中心
- 策略管理
- 租户隔离
- 审计监控
- 运维控制

#### 5) 外部协同与数据支撑层
- 主数据服务
- 外部能力适配
- 结果存档
- 指标分析

### 典型核心闭环

`多方请求接入 → 平台路由/鉴权 → 中心能力处理 → 结果返回/分发`

### 视觉重点

- 中心能力层应最突出
- 平台治理层应清楚但不压过中心能力层
- 外部系统要明确表达为依赖，不要伪装成内部核心功能

---

## 模板 3：运营治理型系统模板

### 适用场景

适用于系统的核心价值不只是执行业务，而是提供治理、控制、审计、运营分析能力。

例如：
- 风险治理平台
- 合规审计平台
- 运维控制平台
- 安全策略平台
- 运营分析与干预平台

### 推荐分层

1. 管理与接入层
2. 策略与运营层
3. 控制执行层
4. 审计与可观测层
5. 数据与外部依赖层

### 各层典型节点

#### 1) 管理与接入层
- 管理后台
- 任务接入
- 告警接入
- 策略发布入口

#### 2) 策略与运营层
- 策略管理
- 规则配置
- 运营编排
- 人工干预

#### 3) 控制执行层
- 风险控制
- 审批执行
- 任务控制
- 指令执行

#### 4) 审计与可观测层
- 审计留痕
- 操作追踪
- 监控告警
- 指标统计

#### 5) 数据与外部依赖层
- 主数据读取
- 外部事件接入
- 结果归档
- 报表输出

### 典型核心闭环

`策略配置/事件触发 → 控制执行 → 审计记录 → 指标反馈 → 运营调整`

### 容易犯的错

- 把“监控告警”画成唯一中心，导致系统目标变成运维系统
- 把审计和运营分析漏掉，导致治理型系统失去治理闭环

---

## 模板 4：任务处理/调度驱动型系统模板

### 适用场景

适用于系统的核心价值来自任务编排、调度、执行、回收与追踪。

例如：
- 调度平台
- 批处理平台
- 自动化执行平台
- 工单流转平台
- Agent/Workflow 执行平台

### 推荐分层

1. 任务入口层
2. 编排与调度层
3. 执行能力层
4. 控制与追踪层
5. 数据与外部协同层

### 各层典型节点

#### 1) 任务入口层
- 任务创建
- 批量导入
- API 提交
- 定时触发

#### 2) 编排与调度层
- 流程编排
- 优先级调度
- 队列分发
- 重试协调

#### 3) 执行能力层
- 执行器管理
- 任务执行
- 结果处理
- 状态更新

#### 4) 控制与追踪层
- 运行监控
- 审计日志
- 告警通知
- 人工干预

#### 5) 数据与外部协同层
- 任务存档
- 外部系统调用
- 结果回传
- 指标分析

### 典型核心闭环

`任务进入 → 编排调度 → 执行处理 → 状态回写/结果回传`

### 评审提醒

如果图里已经开始大量表现“失败后重试第几次、超时后进入哪个分支”，那已经从功能架构图滑向流程图/状态图了。

## 模板 5：数据服务/数据加工型系统模板

### 适用场景

适用于系统核心价值来自数据采集、加工、服务化输出，但当前仍然是“功能架构图”而不是技术数据链路图。

例如：
- 指标平台
- 数据服务平台
- 标签平台
- 数据质量平台
- 数据产品服务系统

### 推荐分层

1. 数据接入层
2. 数据处理与编排层
3. 核心数据能力层
4. 数据治理与服务层
5. 外部消费与反馈层

### 各层典型节点

#### 1) 数据接入层
- 数据采集
- 消息接入
- 批量导入
- 外部数据同步

#### 2) 数据处理与编排层
- 清洗转换
- 任务编排
- 数据校验
- 加工调度

#### 3) 核心数据能力层
- 指标计算
- 标签生成
- 画像构建
- 结果聚合

#### 4) 数据治理与服务层
- 元数据管理
- 质量校验
- 权限控制
- 数据服务发布

#### 5) 外部消费与反馈层
- 查询服务
- 结果分发
- 业务消费接口
- 反馈修正

### 典型核心闭环

`数据接入 → 加工编排 → 核心数据能力产出 → 服务化输出`

### 特别注意

在这个模板中：
- 可以表达“数据能力”
- 但不要下沉成 Spark/Flink/Kafka/Hive 等具体技术名词
- 否则会从功能架构图变成技术架构图

---

## 模板 6：多角色协同业务系统模板

### 适用场景

适用于多个角色参与、多个功能域协作、需要突出业务职责分区的系统。

例如：
- 交易协同平台
- 履约协同系统
- 客服协同系统
- 采购/审批/履约联动系统

### 推荐分层

1. 角色交互层
2. 场景应用层
3. 领域能力层
4. 协同与治理层
5. 数据与外部接口层

### 各层典型节点

#### 1) 角色交互层
- 用户入口
- 审批入口
- 运营入口
- 管理入口

#### 2) 场景应用层
- 下单场景
- 审批场景
- 履约场景
- 运营场景

#### 3) 领域能力层
- 订单能力
- 库存能力
- 支付能力
- 风控能力
- 通知能力

#### 4) 协同与治理层
- 规则中心
- 权限控制
- 审计记录
- 异常处理

#### 5) 数据与外部接口层
- 主数据接口
- 第三方接口
- 结果归档
- 指标服务

### 典型核心闭环

`角色发起 → 场景应用编排 → 多领域能力协作 → 结果反馈`

### 适配建议

如果领域边界是主要难点，应搭配 DDD 类 skill 先把领域/上下文边界想清楚，再套这个模板。

## 模板 7：轻量工具/内部效率系统模板

### 适用场景

适用于功能相对集中、系统规模不大，但仍需要有清晰功能边界和治理能力的内部工具型系统。

例如：
- 配置工具
- 审批辅助工具
- 运营后台工具
- 报表与导出工具

### 推荐分层

1. 入口层
2. 功能处理层
3. 管理与治理层
4. 数据与集成层

### 各层典型节点

#### 1) 入口层
- 后台入口
- 工具入口
- API 入口

#### 2) 功能处理层
- 数据处理
- 查询分析
- 批量操作
- 结果导出

#### 3) 管理与治理层
- 配置管理
- 权限管理
- 操作审计
- 监控告警

#### 4) 数据与集成层
- 数据读取
- 外部接口
- 结果存档

### 典型核心闭环

`入口触发 → 功能处理 → 输出结果`

### 注意

轻量系统也不要把所有功能堆在一个“大模块”里。

即使简单，也应该至少区分：
- 面向用户的入口
- 实际功能处理
- 管理治理
- 数据协同

## 模板选型决策表

### 问题 1：系统最核心的价值来自哪里？

- 用户/业务操作驱动 → 模板 1 或 模板 6
- 平台统一能力输出 → 模板 2
- 治理/审计/控制 → 模板 3
- 调度/执行/任务闭环 → 模板 4
- 数据加工/服务化 → 模板 5
- 小型内部效率工具 → 模板 7

### 问题 2：需要重点突出什么？

- 功能分层本身 → 模板 1
- 中心平台能力 → 模板 2
- 治理闭环 → 模板 3
- 执行闭环 → 模板 4
- 数据能力产出 → 模板 5
- 多角色与多领域协作 → 模板 6
- 低复杂度清晰边界 → 模板 7

### 问题 3：最容易画错成什么图？

- 模板 1：容易错成技术架构图
- 模板 2：容易错成系统集成关系图
- 模板 3：容易错成监控告警拓扑图
- 模板 4：容易错成流程图/状态图
- 模板 5：容易错成数据技术栈图
- 模板 6：容易错成业务流程图
- 模板 7：容易错成页面菜单图

## 快速裁剪规则

如果模板太重，不要重画，按下面方式裁剪：

- 五层裁成四层：合并“应用编排层”和“核心能力层”
- 五层裁成三层：保留“入口层 / 核心能力层 / 支撑协同层”
- 平台型系统：弱化入口层，突出中心能力层
- 治理型系统：强化治理闭环，但不要替代核心功能路径
- 数据型系统：保留数据能力语义，去掉具体技术实现名词

---

## 禁止事项

无论套哪个模板，都禁止：

- 把部署节点、机器、机房、容器编排拓扑混进来
- 把 Kafka / Redis / MySQL / Flink 之类具体技术产品当成功能节点主体
- 把详细时序、异常分支、重试分支塞进功能架构图
- 把页面菜单结构误当成功能分层
- 把“其他”“通用”“基础模块”当成偷懒容器
- 同一层同时出现“订单处理”和“Kubernetes”这类不同抽象级别内容

---

## 最终自检

在输出之前，快速检查：

- 这张图是否清楚回答了系统目标？
- 是否能一眼看出功能分层逻辑？
- 是否能识别核心功能闭环？
- 是否把支撑/治理能力放在正确位置？
- 是否避免滑向技术架构、流程图、部署图？
- 是否仍然保留足够信息来指导后续系统架构与详细设计？
