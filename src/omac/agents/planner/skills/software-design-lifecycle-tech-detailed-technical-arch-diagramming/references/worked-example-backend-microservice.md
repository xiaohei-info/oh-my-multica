# Worked Example: 后端微服务型技术架构图

## 1. 适用场景

Use this example when the system is primarily:
- request/response driven
- transaction-heavy
- service-layered
- dependent on gateway + application services + durable stores + async side processing

Typical domains:
- 交易/订单
- 支付/账务
- 风控接入
- B2B/B2C 平台后端
- 管理后台 + 核心业务服务

This example is intentionally generic. It teaches **how to structure the technical architecture artifact**, not how to hardcode one business domain.

## 2. 这类技术架构图必须回答什么

A reviewer should quickly see:
- 请求从哪里进入
- 哪些服务同步协作完成主事务
- 哪些能力通过异步链路解耦
- 哪些存储分别承载事务、缓存、审计、搜索或消息
- 稳定性靠哪些结构保障
- 哪些外部依赖要按不可靠依赖处理

## 3. 推荐视角与布局

### 推荐视角
- 主图：**逻辑视图 + 运行时视图**
- 补图可选：
  - 数据视图
  - 安全视图
  - 部署视图

### 推荐布局
- 主布局：**分层堆叠（Tier Stack）**
- 局部混合：
  - 如果有统一接入/注册/服务治理中心，可在局部使用 **Hub-Spoke**

### 推荐层次
1. 接入/交互层
2. 控制/治理层
3. 执行/计算层
4. 数据/资源层
5. 观测/安全/基础设施侧带

## 4. 节点映射模板

### 接入/交互层
- 用户端 / 上游调用方 → 外部依赖颜色，云形或矩形
- API Gateway / BFF / LB → 接入层颜色，菱形或六边形

### 控制/治理层
- 配置中心 → 控制层颜色，椭圆或文档形
- 注册发现 / 服务治理 → 控制层颜色，椭圆或菱形
- 调度器 / 任务编排器 → 控制层颜色，矩形
- 规则引擎 / 审批引擎 → 控制层颜色，矩形/立方体

### 执行/计算层
- 核心应用服务 → 执行层颜色，矩形/六边形
- Worker / Async Processor → 执行层颜色，紧凑矩形阵列
- Adapter / Anti-Corruption Layer → 执行层颜色，圆角矩形
- Sidecar / 安全代理 / 服务网格代理 → 小型六边形或矩形

### 数据/资源层
- 主事务库 → 数据层颜色，圆柱体
- 缓存 → 数据层颜色，椭圆
- 消息队列 / 事件总线 → 数据层颜色，椭圆或管道
- 搜索/索引存储 → 数据层颜色，圆柱体
- 对象存储 / 附件存储 / 归档存储 → 数据层颜色，圆柱体
- 审计日志 / 事件历史 → 数据层颜色，圆柱体

### 观测与安全侧带
- Metrics / Trace / Log → 基础设施色
- 告警中心 → 基础设施色
- IAM / KMS / 审计点 → 安全色

## 5. 主链路建模方法

先只画主事务成功路径，例如：
1. 外部请求进入入口网关
2. 网关做鉴权/限流/路由
3. 应用编排服务聚合/校验请求
4. 核心领域服务执行业务逻辑
5. 事务数据落主库
6. 返回同步结果
7. 同时发布异步领域事件给下游处理链

### 主链路的推荐线型
- 外部 → 网关：实线实心箭头
- 网关 → 应用服务：实线实心箭头
- 应用服务 → 核心服务：实线实心箭头
- 核心服务 → 主库：实线空心箭头或普通实线
- 核心服务 → MQ/事件总线：虚线箭头或实线空心箭头，视“事务后发布”还是“同步写入中间件”而定

### 推荐标注格式
- `鉴权路由 / HTTPS`
- `同步调用 / RPC`
- `写入事务 / SQL`
- `发布事件 / Async`

## 6. 异步侧链建模方法

微服务型技术架构图常见错误是：
- 只画同步请求主链路
- 把异步消费链漏掉
- 导致人看不出系统真实负载和稳定性结构

至少要画出：
- 事件发布方
- 消费 Worker
- 下游写入/通知/索引/审计路径
- 是否重试
- 是否死信

推荐结构：
- 主服务 → MQ/事件总线：虚线箭头 `发布事件 / Async`
- MQ → Worker Pool：虚线箭头 `订阅消费 / Consumer Group`
- Worker Pool → 搜索/审计/通知系统：虚线箭头或实线空心箭头
- Worker Pool ↻ MQ：回环箭头 `重试 3次 / 指数退避`
- Worker Pool → 死信队列：红色虚线 `失败转移 / DLQ`

## 7. 稳定性如何画出来

### 7.1 横向扩展
- 核心无状态服务并列多个实例
- 标注 `副本 N` 或 `可横向扩展`

### 7.2 读写分离
- 主事务库旁并列读副本
- 从查询服务指向读副本
- 从写服务指向主库

### 7.3 缓存降压
- 查询服务 → 缓存：实线空心箭头 `读取热点 / Cache`
- 缓存 → 主库：虚线空心箭头 `失效回源 / Cache-Aside`

### 7.4 限流与熔断
- 在网关或调用边上标注：
  - `限流 / Token Bucket`
  - `熔断 / Fail Fast`
  - `降级 / 默认结果`
- 降级路径用红色虚线旁路

### 7.5 幂等与重试
- 在消费 Worker 或写入服务旁加注释：
  - `幂等键校验`
  - `重试 3次 / 指数退避`
  - `超过阈值 → DLQ`

### 7.6 观测
- Metrics/Trace 节点用虚线指向核心服务
- 告警中心与关键节点间加虚线 `异常事件 / 阈值告警`

## 8. 安全与合规如何画出来

至少显式画出：
- 身份源 / IAM
- 网关鉴权边界
- 敏感数据存储
- 审计日志路径
- 跨域访问或特权访问点

推荐：
- 外部域与内部域之间用边界框区分
- 涉敏链路用红色粗虚线标注 `mTLS / 审计`
- 审计日志与业务主库分开画，不要混成一个“数据库”

## 9. 这类图常见拆图策略

若单图过载，优先拆成：

### 图 1：总体技术架构图
回答：
- 技术栈组合
- 同步/异步主协作关系
- 主存储/控制面/执行面

### 图 2：运行时异步处理图
回答：
- MQ / Worker / Retry / DLQ / 补偿链路

### 图 3：安全与治理图
回答：
- 鉴权、审计、配置、规则、隔离、租户边界

### 图 4：数据视图
回答：
- 事务库、缓存、索引、审计、对象存储的关系

## 10. Draw.io 低保真骨架指令

建议先在 Draw.io 中按分层方式摆放：
- 接入层：`外部调用方`、`入口网关`
- 应用层：`应用编排服务`、`核心领域服务`
- 数据与异步层：`主事务库`、`分布式缓存`、`事件总线`、`异步Worker池`、`检索索引`、`审计日志库`、`死信队列`
- 观测层：`监控/追踪`、`告警中心`

建议图元与线型：
- `入口网关` 用菱形或网关样式
- `主事务库/检索索引/审计日志库` 用圆柱体
- `事件总线/死信队列/缓存` 用总线/椭圆/胶囊样式区分异步与热点层
- 实线：HTTPS / 鉴权请求、RPC / 路由、同步调用、写入事务、读取热点、写索引、写审计
- 虚线：发布事件、订阅消费、失效回源、失败转移、监控/告警信号

注意：
- 这只是低保真骨架，用来先确认层次、主同步链和异步补充链
- 实际出图时仍必须回到 full spec 应用颜色、形状、线宽、图例、布局、标注和反模式检查

## 11. 评审时必须追问的问题

- 主事务成功依赖哪些同步调用？
- 哪些链路可以异步化？为什么？
- 哪些状态必须强一致？哪些可以最终一致？
- 外部依赖失败后如何处理？
- 是否存在单点执行器、单点存储、单点控制面？
- 重试是否会放大流量风暴？
- 审计、监控、回放、人工介入入口在哪里？
- 项目结构与运行结构是否一致到足以维护？若不一致，是否需补项目架构图？

## 12. 本示例最容易犯的错

1. 把所有服务画成同一排矩形，完全看不出层次
2. 把缓存、索引、审计、主事务库都画成一个“数据库”
3. 完全忽略 MQ / Worker / Retry / DLQ
4. 不画观测与告警路径
5. 不区分同步强依赖与异步弱依赖
6. 用部署细节淹没技术实现逻辑
7. 只标中间件名字，不标动作与协作语义

## 13. 何时不该套用本示例

不要硬套本示例，当系统本质更像：
- 流水线 ETL / AI 推理 → 用 Pipeline 示例
- 零信任网络与安全分域主导 → 用 Security / Concentric 示例
- 强控制平面 + 多卫星节点 → 用 Hub-Spoke 示例
- 去中心化 P2P / mesh → 用 Mesh 示例
