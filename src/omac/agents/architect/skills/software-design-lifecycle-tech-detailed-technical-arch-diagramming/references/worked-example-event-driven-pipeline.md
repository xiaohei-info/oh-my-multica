# Worked Example: 事件驱动 / 异步处理型技术架构图

## 1. 适用场景

Use this example when the system is fundamentally shaped by:
- event ingestion
- asynchronous fan-out
- staged processing
- pipeline or stream semantics
- retries, backpressure, or replay as first-class concerns

Typical domains:
- 风控事件处理
- 数据采集/清洗/分发
- 通知中心
- 工作流自动化
- AI/ML 推理流水线
- 日志/埋点/行为分析平台

## 2. 这类技术架构图必须回答什么

A reviewer should quickly see:
- 事件从哪里进入
- 被什么通道接住
- 经过哪些阶段处理
- 哪些阶段并行，哪些阶段串行
- 哪些阶段是同步配置/控制，哪些阶段是真实执行
- 出错时如何重试、积压、隔离、回放
- 如何监控 backlog、延迟、丢失、乱序和失败率

## 3. 推荐视角与布局

### 推荐视角
- 主图：**运行时视图 + 数据视图**
- 补图可选：
  - 逻辑视图
  - 安全视图
  - 部署视图

### 推荐布局
- 主布局：**流水线/链式（Pipeline）**
- 可局部混合：
  - 中央事件总线可局部用 **Hub-Spoke**

### 典型阶段
1. 采集/接入
2. 缓冲/解耦
3. 解析/校验
4. 路由/分类
5. 处理/计算
6. 落库/索引/通知/回放
7. 观测/补偿/治理侧带

## 4. 节点映射模板

### 接入阶段
- SDK / Agent / Gateway / Webhook Receiver / Upload Endpoint
- 常用形状：矩形、六边形、云形

### 缓冲与通道阶段
- MQ / Event Bus / Stream Topic / Buffer Queue
- 常用形状：椭圆或管道

### 处理阶段
- Parser / Validator / Router / Enricher / Policy Engine / Inference Worker
- 常用形状：矩形或立方体
- 并发执行器建议用虚线框包住多个小矩形阵列

### 状态与结果阶段
- 原始事件存储
- 处理结果库
- 搜索索引
- 特征/缓存存储
- 死信队列 / 回放队列

### 治理与运维阶段
- 调度器 / Workflow Engine
- 配置中心 / Schema Registry
- 告警中心 / Trace / Metrics
- 审计与重放控制台

## 5. 主路径建模方法

这类系统最重要的是：
- 让人一眼看懂“事件是如何穿过各阶段被处理的”
- 而不是把它画成微服务 RPC 图

推荐主链路：
1. 事件接入
2. 写入缓冲/总线
3. 消费解析
4. 校验/规范化
5. 路由到不同处理分支
6. 执行目标处理
7. 落结果 / 发通知 / 写审计 / 写索引

### 推荐线型
- 主数据推进：粗实线或实线空心箭头
- 事件投递/消费：虚线箭头
- 控制/配置下发：实线空心箭头或点划线
- 回放/补偿/重试：回环箭头或红色虚线旁路

### 推荐标注
- `写入事件 / Append`
- `订阅消费 / Consumer`
- `解析规范化 / Transform`
- `路由分发 / Rule Match`
- `异步处理 / Worker Pool`
- `失败回放 / Replay`

## 6. 这类图最核心的技术语义

### 6.1 并发度
对每个处理阶段，尽量说明：
- 单实例还是多实例
- 并发度是否可扩展
- 是否按 partition/key 有序消费

可视化方式：
- 虚线框中并列多个执行器
- 标注 `并发度 N`
- 标注 `按 Key 有序` / `分区消费`

### 6.2 背压与积压
事件系统不应只画 happy path。
必须至少有一个位置能看出：
- 队列积压监控
- 消费速度与生产速度不匹配时的缓冲点
- 超阈值告警或限流点

可视化方式：
- 队列旁标注 `Backlog Watch`
- 调度/治理节点虚线指向通道 `消费调节 / Pause / Resume`

### 6.3 重试与死信
推荐标准结构：
- 主消费器 ↻ 回环 `重试 3次 / 指数退避`
- 失败超过阈值 → `DLQ`
- DLQ → 人工处理/重放控制台

### 6.4 回放与幂等
事件系统常见关键点：
- 是否可回放
- 回放的输入源在哪
- 幂等边界在哪

可视化方式：
- 事件历史库 / 原始事件存储
- Replay 控制台 → 回放队列 / 事件总线
- 处理节点旁注：`幂等键校验`

### 6.5 顺序与乱序
如果业务对顺序敏感，必须显式表达：
- 全局无序但 Key 内有序
- 同一实体串行处理
- 乱序容忍与纠偏机制

可通过：
- 连线标注
- Callout 注释框
- Router/Partition 节点说明

## 7. 稳定性如何画出来

### 7.1 削峰填谷
- 接入点 → 队列/总线
- 明确缓冲语义：`削峰缓冲 / Queue`

### 7.2 处理隔离
- 不同处理分支分开画
- 高风险分支或慢分支放到独立泳道/虚线框
- 标注 `隔离消费组` / `独立 Worker Pool`

### 7.3 热点与扩缩容
- 某一阶段可能成为热点
- 并列实例 + 标注 `可水平扩展`
- 必要时说明 `分区数 M / Consumer Group N`

### 7.4 可观测性
至少显式出现：
- backlog
- 消费延迟
- 失败率
- 重试率
- DLQ 规模
- 关键链路 trace

### 7.5 降级策略
事件系统的降级不一定是“返回默认值”，更常见的是：
- 暂停非关键消费者
- 保留原始事件稍后处理
- 降级为仅审计不执行
- 切人工处理

这些都应通过旁路或治理节点表现。

## 8. 安全与治理如何画出来

若涉及外部事件接入或高风险数据，建议显式画出：
- schema registry / contract registry
- webhook 签名校验 / source auth
- 多租户隔离带
- 敏感字段脱敏或 redaction 节点
- 审计与回放权限边界

不要让任何人误以为“能接入事件 = 能随意重放事件”。

## 9. Draw.io 低保真骨架指令

建议先在 Draw.io 中按以下骨架落位，再进入正式样式细化：
- 第一列：`事件来源`、`接入网关`
- 第二列中央：`事件总线`
- 第三列：`解析/规范化`、`路由分类`
- 第四列并行：`处理Worker池 A`、`处理Worker池 B`
- 第五列：`结果库`、`检索索引`、`通知适配器`
- 侧边异常/运维链：`重试队列`、`死信队列`、`重放控制台`、`监控/追踪`

建议连线语义：
- 实线：写入事件、校验通过、写结果、写索引、发送通知、回放
- 虚线：发布事件、订阅消费、失败重试、再次消费、超过阈值、监控信号
- `路由分类` 使用菱形；`事件总线/重试队列/死信队列` 用总线或椭圆/胶囊；存储用圆柱体

注意：
- 低保真骨架只负责先把主链路、异常链路和观测链路摆对
- 实际正式出图仍需回到 full spec 进行颜色、线型、图例、布局密度、注释和反模式检查

## 10. 评审时必须追问的问题

- 是否存在事件风暴或下游慢消费拖垮整体的风险？
- 每个阶段的并发度和顺序语义是什么？
- 哪些阶段必须幂等？
- 失败后是重试、旁路、死信还是人工接管？
- 是否保留原始事件，能否重放？
- 事件 schema 如何演进、兼容、回滚？
- backlog / retry / DLQ / lag 的告警在哪里？
- 哪些分支应独立隔离，避免一个慢消费者影响全部链路？

## 11. 本示例最容易犯的错

1. 把事件系统画成同步 RPC 服务链
2. 不画 queue/topic/buffer，只画处理服务
3. 不区分主总线、重试队列、死信队列
4. 不画 replay 能力
5. 不表达并发度、顺序性、分区语义
6. 不画 backlog / lag / retry / DLQ 的观测链路
7. 所有分支同权重，导致看不出核心链路

## 12. 何时不该套用本示例

不要硬套本示例，当系统本质更像：
- 经典请求响应事务后端 → 用 backend microservice 示例
- 单体/分层业务平台 → 用 Tier Stack 更纯的服务型图
- 强安全域隔离架构 → 用 Security / Concentric 视角
- 纯项目结构/代码模块组织 → 另补项目架构图
