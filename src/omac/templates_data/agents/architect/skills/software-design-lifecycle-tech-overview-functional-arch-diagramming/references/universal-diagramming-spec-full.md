# Skill：通用功能架构图绘制（Universal Architecture Diagramming）

**版本**：v2.0  
**性质**：工具无关、场景无关的元规范（Meta-Spec）  
**核心目标**：在任何系统设计场景中，用一致的视觉语法讲清楚「有什么、分几类、怎么连、谁依赖谁」。  
**目标受众**：产品经理、研发工程师、架构师  
**回答的核心问题**：系统建设目标是什么？包含哪些功能/组件？这些实体如何分层/分类与协作？

---

## 1. 元模型：三种抽象元素

任何架构图只由以下三类元素构成，不绑定任何业务名词：

| 元素 | 抽象定义 | 视觉表现 |
|-----|---------|---------|
| **容器（Container）** | 逻辑边界，用于分组或分层 | 矩形框，可实线/虚线/圆角，可带背景色 |
| **节点（Node）** | 架构中的实体（功能、服务、组件、数据、硬件等） | 矩形/圆角矩形/圆柱/菱形等，内部文字 |
| **边（Edge）** | 实体间的关系（依赖、流向、调用、包含） | 实线/虚线/箭头，可带文字标签 |

**结构铁律**：
- 容器可以嵌套（容器内放容器），节点只能放在最底层容器内。
- 边只能连接节点，禁止边直接连接容器。
- 架构图最多 2 层容器（如：层 → 组），更深细节放到子图。

## 2. 视觉编码系统（可配置）

本 Skill 不提供固定配色，而是提供一套视觉编码协议。绘制前必须为当前场景分配编码。

### 2.1 色彩语义（Color Semantics）

选择 **1 个主色系 + 2–4 个辅助色**，总颜色数 ≤ 6。

| 编码位 | 建议用途（根据场景自选） | 色值策略 |
|-------|------------------------|---------|
| **主色** | 核心实体 / 本系统主体 | 品牌色或中性蓝 |
| **辅色 A** | 外部系统 / 第三方依赖 | 灰色系 |
| **辅色 B** | 数据层 / 存储 / 持久化 | 绿色系或紫色系 |
| **辅色 C** | 控制面 / 配置 / 治理 | 橙色系或黄色系 |
| **辅色 D** | 危险区 / 安全 / 审计（可选） | 红色系 |
| **背景/容器** | 分组底色 | 对应颜色的 5%–10% 透明度 |

**配色铁律**：
- 节点背景必须是浅色低饱和（确保黑字可读），边框用对应色的高饱和版本。
- 同一容器内的节点应共享同一色系，通过边框深浅区分主次。
- 禁止用颜色表达「重要性」或「优先级」——重要性通过位置（靠近中心/上层）或边框粗细体现。

### 2.2 形状语义（Shape Semantics）

| 形状 | 表达的实体类型 |
|-----|------------------|
| 圆角矩形 | 通用功能模块、微服务、业务组件（最常用，默认） |
| 直角矩形 | 外部系统、基础设施、硬件资源 |
| 圆柱 | 数据库、数据仓库、存储、缓存 |
| 菱形 | 决策点、网关、负载均衡、过滤器 |
| 圆形 | 端点、客户端、用户、外部触发源 |
| 云形 | 云平台、SaaS、第三方服务 |
| 文档形 | 配置、策略、文档、模板 |

### 2.3 线型语义（Line Semantics）

| 线型 | 表达的关系 |
|-----|-----------|
| 实线 + 单向箭头 | 强依赖、数据流向、调用方向 |
| 实线 + 双向箭头 | 双向通信、同步调用 |
| 虚线 + 箭头 | 弱依赖、异步消息、事件触发、未来规划 |
| 虚线 + 无箭头 | 包含关系、逻辑归属、同一分组 |
| 粗实线 | 主链路、核心数据流 |

## 3. 布局模式库（根据架构类型选择）

绘制前必须选定以下模式之一：

### 模式 A：分层拓扑（Layered）
- **适用**：功能架构、分层技术架构（如展示层/业务层/数据层）。
- **布局**：纵向堆叠容器，自上而下或自下而上表示依赖方向。
- **规则**：上层节点指向下层节点表示依赖；同层节点不画纵向箭头。
- **示例**：B/S 系统功能架构、OSI 网络模型、DDD 分层架构。

### 模式 B：服务网格/矩阵（Matrix）
- **适用**：微服务架构、中台架构、能力地图。
- **布局**：二维矩阵，横向为业务域，纵向为能力类型；或平铺网格。
- **规则**：节点均匀分布，用虚线框分组；节点间用轻量箭头表示调用。
- **示例**：电商中台（商品域/订单域/支付域 × 数据服务/业务服务/接口服务）。

### 模式 C：中心辐射（Hub-and-Spoke）
- **适用**：网关架构、总线架构、平台型产品架构。
- **布局**：中心放置核心节点（网关/总线/平台），四周辐射子系统。
- **规则**：所有边从外围指向中心，或从中心指向外围，保持方向一致。
- **示例**：API 网关架构、ESB 企业服务总线、开放平台。

### 模式 D：管道/流（Pipeline）
- **适用**：数据流架构、ETL 流程、CI/CD 架构、事件驱动架构。
- **布局**：横向或纵向的线性序列，节点按处理顺序排列。
- **规则**：边必须沿主线方向，禁止反向箭头；分支用菱形决策点引出。
- **示例**：实时计算链路、推荐系统数据流、DevOps 流水线。

### 模式 E：混合/自由（Hybrid）
- **适用**：复杂企业级架构、总体架构蓝图。
- **布局**：局部使用不同模式，用明显的留白或分隔线区分区域。
- **规则**：每个局部区域内部遵循单一模式；区域间用粗虚线或标签分隔。
- **示例**：企业级总体技术架构（左侧业务中台用矩阵，右侧基础设施用分层）。

## 4. 构图与间距规范（跨工具通用）

| 规范项 | 标准值 | 说明 |
|-------|-------|------|
| 画布边距 | ≥ 40px | 四边留白，防止裁剪 |
| 容器内边距 | ≥ 20px | 容器边框到内部节点的距离 |
| 容器间距 | ≥ 30px | 同级容器之间 |
| 节点间距 | ≥ 16px（横向）/ ≥ 12px（纵向） | 防止视觉粘连 |
| 节点最小尺寸 | 宽 ≥ 100px，高 ≥ 40px | 确保文字不溢出 |
| 文字边距 | 节点内文字距边框 ≥ 8px | 呼吸感 |
| 边与节点连接点 | 连接在节点边框中点 | 避免从角落出线 |
| 边标签位置 | 箭头中段上方，不跨越其他节点 | 清晰标注关系语义 |

**对齐铁律**：
- 所有节点必须网格对齐（建议 10px 或 20px 网格）。
- 同一容器内的节点必须顶部对齐或中心对齐，禁止随意错落。
- 同类节点宽度建议统一（如统一 160px），高度自适应但同组建议等高。

## 5. 场景适配决策表（绘制前自检）

接到任务时，必须先回答以下问题：

| 问题 | 选项 | 影响 |
|-----|------|------|
| 这张图回答的核心问题是什么？ | A. 有哪些功能？怎么分层？<<br>>B. 有哪些组件？怎么调用？<<br>>C. 数据怎么流？经过哪些处理？<<br>>D. 部署在哪？什么拓扑？<<br>>E. 安全边界在哪？怎么防护？ | 决定布局模式（功能→分层/矩阵，数据→管道，部署→分层/自由，安全→分层/中心辐射） |
| 节点之间最重要的是分类还是连接？ | A. 分类（展示有什么）<<br>>B. 连接（展示怎么交互） | 分类优先则容器和分组要突出，边要克制；连接优先则边和流向要突出 |
| 是否需要表达外部依赖？ | 是 / 否 | 决定是否引入「辅色 A（外部色）」和外部系统形状 |
| 是否需要表达数据持久化？ | 是 / 否 | 决定是否引入圆柱形节点 |
| 是否需要表达控制/治理？ | 是 / 否 | 决定是否引入菱形节点和虚线边 |
| 受众是谁？ | A. 产品经理（业务语言）<<br>>B. 研发工程师（技术语言）<<br>>C. 架构师/CTO（抽象语言）<<br>>D. 客户/管理层（精简语言） | 决定节点命名粒度、技术术语深度、图的复杂度 |

## 6. 命名与粒度规范

### 6.1 节点命名

- **产品经理场景**：使用「业务动作 + 对象」，如「订单创建」「库存扣减」，禁止用「ServiceA」「ModuleX」。
- **研发场景**：使用「技术类型 + 职责」，如「订单服务」「支付网关」「用户缓存」，禁止用纯动词或纯名词（如「处理」「数据库」）。
- **架构师场景**：使用「领域 + 模式」，如「订单限界上下文」「支付防腐层」。
- **通用铁律**：节点名控制在 **2–6 个中文词** 或 **1–3 个英文单词**，禁止换行（除非节点宽度 > 200px）。

### 6.2 容器/分组命名

- 必须是分类维度而非罗列，如「核心域」「基础设施」「第三方集成」「数据持久化」。
- 禁止用「其他」「miscellaneous」「未分类」作为分组名——若出现此类分组，说明抽象层级有误，需重新归类。

## 7. 反模式与禁忌（任何场景均禁止）

| 反模式 | 错误表现 | 正确做法 |
|-------|---------|---------|
| **名词泄露** | 图中出现特定公司名、产品名、版本号 | 使用抽象命名，如「消息队列」而非「Kafka 3.2」 |
| **形状混用无语义** | 圆角矩形和直角矩形随机使用 | 选定形状词典后全图一致，并在角落添加图例 |
| **箭头泛滥** | 每两个节点之间都有箭头 | 只画「强依赖」和「主链路」箭头，弱依赖用虚线或省略 |
| **颜色表达优先级** | 用红色表示「重要」，黄色表示「次要」 | 优先级通过位置或边框粗细表达 |
| **容器过深** | 容器嵌套超过 3 层 | 架构图最多 2 层容器（层 → 组），细节放到子图 |
| **混合粒度** | 同一容器内既有「用户服务」又有「MySQL 连接池配置」 | 同容器节点必须是同一抽象级别，不同级别要分层 |
| **无留白** | 节点填满整张画布，没有呼吸空间 | 严格遵守间距规范，保持 40px 以上边距 |

## 8. 工具适配说明（Tool-Agnostic）

本 Skill 为纯语义规范，可自行根据场景选择以下工具绘制：

| 工具 | 适用场景 | 映射要点 |
|-----|---------|---------|
| **Excalidraw** | 手绘风格、快速协作、评审会议 | Rectangle + Dashed stroke 做容器；手动输入 Hex 配色；Grid + Snap 对齐 |
| **Draw.io (diagrams.net)** | 正式文档、精确对齐、印刷级输出、可编辑源文件沉淀 | Container / Swimlane 做容器；Connector 画边；Arrange → Align 做对齐；保留 `.drawio` 源文件 |
| **SVG / HTML** | 网页嵌入、动态交互、数据驱动 | `<rect>` + `stroke-dasharray` 做容器；`<line>`/`<path>` + marker 做边；手动计算坐标实现精确布局 |

**执行原则**：无论使用哪种工具，绘制前必须声明「采用通用架构图元 Skill，布局模式为 [X]，主色为 [Y]」，以确保输出一致性。

## 9. 执行流程（Step-by-Step）

**Step 1：场景自检**
回答第 5 节决策表的全部问题，确定布局模式、配色方案、形状词典、受众语言。

**Step 2：语义抽象**
将系统实体清单抽象为「节点」，将逻辑边界抽象为「容器」，将关系抽象为「边」。

**Step 3：画布规划**
- 预估节点数量：≤ 8 个用 4:3；8–20 个用 16:9；> 20 个考虑横向长图或分页。
- 预留右侧/底部空间给图例和说明。

**Step 4：骨架搭建**
按选定的布局模式，先画出所有容器（层/组/域），标注容器标题，不填充节点。

**Step 5：节点排布**
将节点放入最底层容器，按矩阵或序列均匀分布，确保网格对齐和间距规范。

**Step 6：边与标签**
绘制节点间的边，只保留强依赖和主链路。为关键边添加文字标签说明关系语义。

**Step 7：视觉校验**
- 检查配色是否 ≤ 6 种，节点背景是否为浅色。
- 检查层/容器间距是否 ≥ 30px，节点间距是否 ≥ 16px。
- 检查文字是否全部居中，是否存在溢出。
- 检查是否违反第 7 节任何禁忌。

**Step 8：工具输出**
将校验通过的架构图映射到选定工具（Excalidraw / Draw.io / SVG）的具体语法或操作中。

## 10. 输出物质量标准（Checklist）

一份合格的架构图必须满足：

- [ ]  viewer 能在 10 秒内识别出系统的**建设目标**（通过容器命名和节点分布）
- [ ]  viewer 能在 30 秒内理解**实体分类逻辑**（通过容器分组和颜色语义）
- [ ]  不存在孤立的节点（每个节点至少属于一个容器）
- [ ]  颜色不超过 6 种，节点背景为浅色低饱和
- [ ]  图中没有具体的产品名、公司名、版本号
- [ ]  边只表达强依赖，没有箭头泛滥
- [ ]  容器嵌套不超过 2 层
- [ ]  全图有统一的形状词典和线型语义，并在角落或底部提供图例
